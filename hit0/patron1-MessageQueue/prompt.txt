CONTEXTO DEL PROYECTO
=====================
Estoy desarrollando el Patrón 1 (Message Queue punto a punto) del HIT #0 de un 
Trabajo Práctico universitario sobre Cloud Computing. El repositorio ya existe con 
la estructura TP3_SDyPP/hit0/patron1-MessageQueue. 

OBJETIVO
========
Implementar un sistema producer/consumer con RabbitMQ que demuestre distribución 
de mensajes round-robin entre múltiples consumers. Todo debe correr desde terminal 
sin abrir ningún IDE.

STACK TÉCNICO
=============
- Lenguaje: Python
- Librería RabbitMQ: pika
- Containerización: Docker + Docker Compose
- RabbitMQ: imagen oficial rabbitmq:management

ESPECIFICACIONES TÉCNICAS DETALLADAS
=====================================

producer.py:
- Conectarse a RabbitMQ con retry loop y exponential backoff (hasta 10 intentos, 
  delay 2^i segundos entre intentos)
- Host leído desde variable de entorno RABBITMQ_HOST (default: 'rabbitmq')
- Declarar cola 'task_queue' con durable=True
- Enviar exactamente 10 mensajes con body "Tarea 1" hasta "Tarea 10"
- Cada mensaje con delivery_mode=2 (persistente)
- Imprimir en stdout cada mensaje enviado: "[Producer] Enviado: Tarea X"
- Cerrar conexión al terminar

consumer.py:
- Conectarse a RabbitMQ con el mismo mecanismo de retry que el producer
- Host leído desde variable de entorno RABBITMQ_HOST (default: 'rabbitmq')
- Declarar cola 'task_queue' con durable=True (idempotente si ya existe)
- Identificarse usando socket.gethostname() como CONSUMER_ID
- basic_qos(prefetch_count=1) obligatorio
- Imprimir cada mensaje recibido: "[CONSUMER_ID] Recibido: Tarea X"
- Hacer basic_ack explícito después de imprimir
- Correr indefinidamente con start_consuming()

docker-compose.yml:
- Servicio 'rabbitmq': imagen rabbitmq:management, puertos 5672:5672 y 
  15672:15672, healthcheck con rabbitmq-diagnostics ping cada 10s, 
  timeout 5s, retries 10
- Servicio 'consumer': build con Dockerfile.consumer, depends_on rabbitmq 
  con condition service_healthy, variable de entorno RABBITMQ_HOST=rabbitmq,
  restart: unless-stopped
- Servicio 'producer': build con Dockerfile.producer, depends_on rabbitmq 
  con condition service_healthy, variable de entorno RABBITMQ_HOST=rabbitmq,
  profiles: ["producer"] para que no corra automáticamente con up

Dockerfile.producer y Dockerfile.consumer:
- Base: python:3.11-slim
- WORKDIR /app
- Copiar requirements.txt e instalar dependencias
- Copiar el script correspondiente
- CMD ejecutar el script

requirements.txt:
- pika>=1.3.0

README.md debe incluir:
1. Título: "Patrón 1 - Message Queue (Punto a Punto)"
2. Descripción breve del patrón
3. Diagrama de arquitectura en ASCII mostrando Producer → Queue → Consumer1/Consumer2
4. Sección "Cómo ejecutar" con estos comandos exactos y en este orden:
   a. Levantar RabbitMQ y consumers: docker compose up --scale consumer=2 -d
   b. Esperar healthcheck: docker compose ps (verificar que rabbitmq esté healthy)
   c. Correr producer: docker compose run producer
   d. Ver logs: docker compose logs consumer
   e. Bajar todo: docker compose down
5. Sección "Comportamiento observado" que explique:
   - Con 1 consumer: todos los mensajes van al mismo consumer
   - Con 2 consumers: los mensajes se alternan en round-robin
   - Por qué prefetch_count=1 cambia el comportamiento respecto al default
6. Sección "Decisiones de diseño":
   - durable=True y delivery_mode=2: los mensajes sobreviven reinicio del broker
   - prefetch_count=1: fair dispatch, evita que un consumer acumule mensajes 
     sin procesar mientras otro está libre
   - Exponential backoff en conexión: RabbitMQ tarda ~10-15s en estar listo, 
     sin retry el producer falla inmediatamente
   - Profile "producer" en compose: permite escalar consumers independientemente 
     y disparar el producer manualmente para observar el comportamiento

RESTRICCIONES
=============
- No usar variables hardcodeadas para el host de RabbitMQ, siempre desde env
- No commitear archivos .env (agregar al .gitignore si no existe)
- El sistema debe funcionar ejecutando únicamente comandos docker compose 
  desde la carpeta patron1-MessageQueue, sin instalar pika localmente
- No usar asyncio ni threading, pika bloqueante es suficiente para este caso
