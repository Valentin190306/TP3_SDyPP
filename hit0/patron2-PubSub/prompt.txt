=====================
Estoy desarrollando el Patrón 2 (Pub/Sub Fan-out) del HIT #0 de un

OBJETIVO
========
Implementar un sistema publisher/subscriber con RabbitMQ usando un exchange de tipo
fanout que demuestre que TODOS los suscriptores reciben una copia de cada mensaje
publicado. El caso de uso es una red blockchain donde se notifica a todos los nodos
cuando se mina un nuevo bloque. Todo debe correr desde terminal sin abrir ningún IDE.

STACK TÉCNICO
=============
- Lenguaje: Python
- Librería RabbitMQ: pika
- Containerización: Docker + Docker Compose
- RabbitMQ: imagen oficial rabbitmq:management

ESPECIFICACIONES TÉCNICAS DETALLADAS
=====================================

publisher.py:
- Conectarse a RabbitMQ con retry loop y exponential backoff (hasta 10 intentos,
  delay 2^i segundos entre intentos)
- Host leído desde variable de entorno RABBITMQ_HOST (default: 'rabbitmq')
- Declarar un exchange llamado 'blocks_exchange' de tipo fanout con durable=True
- Publicar exactamente 5 eventos con body JSON:
  {"event": "new_block", "block_number": N, "timestamp": "<ISO timestamp>", "hash": "<uuid4>"}
  donde N va de 1 a 5
- Esperar 1 segundo entre cada publicación (time.sleep(1)) para que los logs sean legibles
- Imprimir en stdout cada evento publicado: "[Publisher] Bloque N publicado: <json>"
- Cerrar conexión al terminar

subscriber.py:
- Conectarse a RabbitMQ con el mismo mecanismo de retry que el publisher
- Host leído desde variable de entorno RABBITMQ_HOST (default: 'rabbitmq')
- Identificarse usando socket.gethostname() como NODE_ID
- Declarar el mismo exchange 'blocks_exchange' de tipo fanout con durable=True
- Crear una cola exclusiva y anónima (queue='', exclusive=True) — se elimina al desconectarse
- Bindear esa cola al exchange 'blocks_exchange'
- NO usar basic_qos ni prefetch (en fan-out no aplica fair dispatch)
- Al recibir un mensaje, parsearlo como JSON e imprimir:
  "[NODE_ID] Nuevo bloque recibido - Bloque N | Hash: <hash> | Timestamp: <timestamp>"
- Hacer basic_ack explícito después de imprimir
- Correr indefinidamente con start_consuming()

docker-compose.yml:
- Servicio 'rabbitmq': imagen rabbitmq:management, puertos 5672:5672 y
  15672:15672, healthcheck con rabbitmq-diagnostics ping cada 10s,
  timeout 5s, retries 10
- Servicio 'subscriber': build con Dockerfile.subscriber, depends_on rabbitmq
  con condition service_healthy, variable de entorno RABBITMQ_HOST=rabbitmq,
  restart: unless-stopped
- Servicio 'publisher': build con Dockerfile.publisher, depends_on rabbitmq
  con condition service_healthy, variable de entorno RABBITMQ_HOST=rabbitmq,
  profiles: ["publisher"] para que no corra automáticamente con up

Dockerfile.publisher y Dockerfile.subscriber:
- Base: python:3.11-slim
- WORKDIR /app
- Copiar requirements.txt e instalar dependencias
- Copiar el script correspondiente
- CMD ejecutar el script

README.md debe incluir:
1. Título: "Patrón 2 - Pub/Sub Fan-out"
2. Descripción breve del patrón y diferencia con el patrón punto a punto
3. Diagrama de arquitectura en ASCII mostrando:
   Publisher → [Exchange fanout: blocks_exchange] → Nodo1 / Nodo2 / Nodo3
4. Sección "Cómo ejecutar" con estos comandos exactos y en este orden:
   a. Levantar RabbitMQ y subscribers: docker compose up --scale subscriber=3 -d
   b. Esperar healthcheck: docker compose ps (verificar que rabbitmq esté healthy)
   c. Correr publisher: docker compose run publisher
   d. Ver logs: docker compose logs subscriber
   e. Bajar todo: docker compose down
5. Sección "Comportamiento observado" que explique:
   - Con 3 subscribers: los 3 reciben TODOS los mensajes (no se reparten, se copian)
   - Diferencia fundamental con el Ejemplo 1: en Message Queue cada mensaje va a UN
     consumer; en Fan-out cada mensaje va a TODOS los suscriptores activos
   - Qué pasa si un subscriber se conecta después de que el publisher ya publicó:
     no recibe los mensajes anteriores porque su cola es exclusiva y temporal
6. Sección "Decisiones de diseño":
   - Exchange fanout con durable=True: el exchange sobrevive reinicios del broker;
     las colas son temporales pero el exchange no
   - Cola exclusiva y anónima por suscriptor: garantiza que cada nodo tenga su
     propia copia independiente de cada mensaje; no se comparte la cola entre nodos
   - Sin prefetch_count: en fan-out no hay competencia entre consumers, cada uno
     tiene su propia cola, el fair dispatch no aplica
   - Profile "publisher" en compose: permite escalar subscribers independientemente
     y disparar el publisher manualmente para observar el comportamiento
   - Payload JSON con uuid4 como hash: simula de forma realista un evento de
     blockchain sin necesidad de implementar criptografía real

RESTRICCIONES
=============
- No usar variables hardcodeadas para el host de RabbitMQ, siempre desde env
- No commitear archivos .env (ya está en el .gitignore del repo)
- El sistema debe funcionar ejecutando únicamente comandos docker compose
  desde la carpeta patron2-PubSub, sin instalar pika localmente
- No usar asyncio ni threading, pika bloqueante es suficiente para este caso
- El publisher NO debe declarar colas, solo el exchange
- Cada subscriber debe crear SU PROPIA cola (exclusive=True), no compartida
